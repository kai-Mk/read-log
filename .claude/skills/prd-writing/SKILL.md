# prd-writing

プロダクト要件定義書（PRD）を作成するスキル。

## 使用可能なツール

- **Read** - ドキュメントを読む
- **Write** - PRDを作成する
- **Glob** - ファイルを探す
- **AskUserQuestion** - ユーザーに確認する

上記以外のツール（Bash、Edit、Grep、WebFetch、WebSearchなど）は使用しない。

## 目的

`docs/ideas`の内容をもとに、開発の指針となるプロダクト要件定義書を作成する。

## 作成の流れ

1. `docs/ideas`内のドキュメントを読み込む
2. アイデアを整理し、PRDの各セクションを埋める
3. 曖昧な部分はユーザーに確認する
4. `docs/product-requirements.md`として出力する

## 各セクションの記述ガイド

### 1. プロダクト概要

#### 1.1 プロダクト名

- プロダクトの正式名称を記載する
- 略称がある場合は併記する

#### 1.2 概要

プロダクトが何をするものかを1-2文で簡潔に記述する。技術的な詳細は含めず、ユーザー視点で書く。

**よい例:**
> 積読本を管理し、読書状況を可視化するWebアプリケーション

**悪い例:**
> React + Honoで作る本の管理システム（技術詳細を含んでいる）

#### 1.3 背景・目的

なぜこのプロダクトを作るのかの動機を記述する。解決したい課題や現状の問題点を明確にし、プロダクトによって達成したいゴールを記載する。

**よい例:**
> 技術書や学術書を購入するが、読む前に次の本を買ってしまい積読が増える。どの本を持っているか把握できず、同じ本を買ってしまうこともある。本の管理と読書状況の可視化によってこの問題を解決する。

**悪い例:**
> 本を管理したいから（動機が不明確で課題が見えない）

### 2. ターゲットユーザー

誰がこのプロダクトを使うのかを具体的に記述する。ユーザーの特徴、前提知識、利用シーンを含める。

**よい例:**
> 技術書や学術書を多く購入するが、積読が増えがちなエンジニアや研究者。本の管理に専用ツールは使っておらず、購入した本を把握しきれていない。

**悪い例:**
> 本を読む人（範囲が広すぎて具体性がない）

### 3. コア機能

プロダクトの中核となる機能を記載する。詳細な仕様は機能設計書で定義する。

各機能について以下を記述する:

- **機能名**: 機能を一言で表す名前を付ける
- **ユーザーストーリー**: 「[ユーザー]として、[目的]のために、[機能]が欲しい」の形式で書く
- **受け入れ条件**: 機能が完成したと判断できる測定可能な条件をリストアップする

**よい例:**
```
### F-01: 本の登録

**ユーザーストーリー**: 積読を管理したいユーザーとして、購入した本を記録するために、本の情報を登録する機能が欲しい

**受け入れ条件**:
- [ ] ISBNを入力すると書籍情報が自動取得される
- [ ] タイトル、著者、表紙画像が表示される
- [ ] 登録した本が一覧に追加される
```

**悪い例:**
```
### F-01: 本の登録

**ユーザーストーリー**: ユーザーとして、本を登録したい

**受け入れ条件**:
- [ ] 簡単に登録できる（「簡単」は測定不能）
- [ ] 使いやすいUIである（「使いやすい」は主観的）
```

### 4. 非機能要件

#### 4.1 パフォーマンス

レスポンスタイムの目標値、同時接続数の想定、データ量の想定を記述する。

**よい例:**
> - ページ読み込みは3秒以内に完了する
> - 同時接続数は10ユーザーを想定する
> - 1ユーザーあたり最大500冊の本を管理できる

**悪い例:**
> - 速くする（目標値がない）

#### 4.2 セキュリティ

認証・認可の要件、データ保護の方針を記述する。今回スコープ外の場合はその旨を明記する。

**よい例:**
> - 認証機能は実装しない（スコープ外）
> - マイ書庫IDはURLに含まれ、IDを知っている人のみアクセス可能とする

**悪い例:**
> - セキュリティに配慮する（具体性がない）

#### 4.3 その他

対応ブラウザ・デバイス、アクセシビリティ要件、国際化対応の有無を記述する。

**よい例:**
> - Chrome、Firefox、Safariの最新版に対応する
> - モバイル対応はスコープ外とする
> - 日本語のみ対応する

**悪い例:**
> - モダンブラウザに対応する（「モダン」の定義が曖昧）

### 5. スコープ外

本プロジェクトで意図的に対応しないことを明記する。将来的に対応する可能性があるものも含める。スコープクリープを防ぐために重要なセクションである。

**よい例:**
> - ユーザー認証・認可
> - モバイルアプリ対応
> - 複数ユーザー間での本の共有機能
> - 読書時間のトラッキング

**悪い例:**
> - 難しい機能（何が難しいか不明）

`template.md` を参照すること。
